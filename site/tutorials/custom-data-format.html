1. Introduction

In this tutorial, you will learn how to create custom Easy Batch components to process data in a non standard format.

The goal is to process recipes defined in the following recipes.txt file. Each recipe is defined on multiple physical records:

  • R record: marks the beginning of a recipe
  • I record: marks an ingredient of the current recipe
  • END record: marks the end of the current recipe, the next recipe (if any) begins on the next line
recipes.txt
R,Easy chocolate cake
I,chocolate,500g
I,sugar,100g
I,cocoa powder,50g
END
R,Ice cream
I,ice,100g
I,cream,50g
END

Each set of physical records representing a recipe should be mapped to the following domain model classes:

public class Recipe {

    private String name;

    private List<Ingredient> ingredients;

    //Getters and setters omitted
}
public class Ingredient {

    private String name;

    private int quantity;

    //Getters and setters omitted
}

To keep the tutorial simple, processing logic should simply write recipes data to the console.

So let's get started!

2. Reading recipes

First, you will define a logical recipe record that represent multiple physical records in the flat file:

public class RecipeRecord extends GenericRecord<Recipe> {

    public RecipeRecord(int number, Recipe recipe) {
        super(number, recipe);
    }
}

This logical record holds the record number and an instance of the recipe as its payload. Next, you will implement a custom RecordReader to read multiple physical records and produce logical RecipeRecord instances, so here is the implementation:

public class RecipeRecordReader implements RecordReader {

    public static final String SEPARATOR = ",";

    /**
     * Data source file.
     */
    private File file;

    /**
     * Scanner to read the input file.
     */
    private Scanner scanner;

    /**
     * The current read record number.
     */
    private int currentRecordNumber;

    public RecipeRecordReader(File file) {
        this.file = file;
    }

    @Override
    public void open() throws Exception {
        scanner = new Scanner(file);
    }

    @Override
    public boolean hasNextRecord() {
        return scanner.hasNextLine();
    }

    @Override
    public Record readNextRecord() throws Exception {
        currentRecordNumber++;
        Recipe recipe = new Recipe();
        while (scanner.hasNext()) {
            String line = scanner.nextLine();
            if (line.startsWith("END")) {
                break;
            } else if (line.startsWith("R")) { //recipe line
                String[] strings = line.split(SEPARATOR);
                recipe.setName(strings[1]);
            } else if (line.startsWith("I")) { //ingredient line
                String[] strings = line.split(SEPARATOR);
                Ingredient ingredient = new Ingredient();
                ingredient.setName(strings[1]);
                ingredient.setQuantity(Integer.valueOf(strings[2]));
                recipe.getIngredients().add(ingredient);
            }
        }
        return new RecipeRecord(currentRecordNumber, recipe);
    }

    @Override
    public Integer getTotalRecords() {
        // not implemented in this tutorial.
        return null;
    }

    @Override
    public String getDataSourceName() {
        return file.getAbsolutePath();
    }

    @Override
    public void close() throws Exception {
        scanner.close();
    }
}

The implementation is pretty straightforward. The devil is in the readNextRecord method: this is where you should iterate on physical records until finding a END record to create a recipe instance.

2. Mapping recipes

The record reader that we implemented not only reads data but also maps data to domain objects. Mapping data to domain objects is the role of the RecordMapper component.

We could have implemented the record reader to only read data and create a RecipeRecord with a String containing physical records to be mapped by a RecipeRecordMapper, but in this case, we would iterate twice on physical records: a first time to read data and a second time to map data. That's why we have implemented the recipe reader to read and map data at the same time, since it is more efficient approach.

Easy Batch provides the GenericRecordMapper that returns the record payload object as is. We can use this mapper to return recipes from the RecipeRecord.

3. Processing recipes

No that you have mapped flat data to domain objects of type Recipe, you will implement a RecipeProcessor that simply writes recipes to the console as required:

public class RecipeProcessor implements RecordProcessor<Recipe, Recipe>  {

    @Override
    public void processRecord(Recipe recipe) throws Exception {
        System.out.println(recipe);
        return recipe;
    }
}

4. Putting it all together

Let's configure an Easy Batch engine to:

  1. Read recipes from recipes.txt file using the custom RecipeRecordReader
  2. Map physical records in flat file to the logical RecipeRecord
  3. Process each recipe using the RecipeProcessor

The following listing provides this configuration:

public class Launcher {

    public static void main(String[] args) throws Exception {

        // Initialize input file recipes.txt
        File recipes = new File(args[0]);

        // Build a batch engine
        Engine engine = new EngineBuilder()
            .reader(new RecipeRecordReader(recipes))
            .mapper(new GenericRecordMapper())
            .processor(new RecipeProcessor())
            .build();

        // Run the batch engine
        engine.call();
    }
}

That's all, you can now run the tutorial and see the results.

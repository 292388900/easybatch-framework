1. Introduction

You have seen in the previous tutorial how to process tweets form a CSV file to compute some statistics. You have filtered undesired data, mapped tweets to the Tweet bean, and validated tweets using the Bean Validation API.

In this tutorial, you will see how to store these tweets into a database. Since only the business logic changes, you will reuse the same application and register a new processor to load validated tweets in the database.

Here is a quick refresh of what you have seen in the previous tutorial:

public class Launcher {

    public static void main(String[] args) throws Exception {

        // Build a batch engine
        Engine engine = new EngineBuilder()
            .reader(new FlatFileRecordReader(new File("tweets.csv")))
            .filter(new HeaderRecordFilter())
            .mapper(new DelimitedRecordMapper(Tweet.class, new String[]{"id", "name", "message"}))
            .validator(new BeanValidationRecordValidator<Tweet>())
            .processor(new TweetCountProcessor())
            .build();

        // Run the batch engine
        Report report = engine.call();

        // Print the batch execution report
        System.out.println("Total tweets containing #EasyBatch: " + report.getBatchResult());
    }
}

We will change the TweetCountProcessor by another implementation to store tweets in a database instead of counting them.

To keep the tutorial simple, a in-memory database will be used.

Tweets will be inserted in the tweet table.

The following implementation of the RecordProcessor interface uses Hibernate APIs to save tweets in the database:

- Tweet table -
Field Type
id INTEGER
user VARCHAR(32)
message VARCHAR(140)
TweetLoader.java
public class TweetLoader implements RecordProcessor<Tweet, Tweet> {

    public Tweet processRecord(Tweet tweet) throws Exception {

        DatabaseUtil.getCurrentSession().saveOrUpdate(tweet);
        return tweet;
    }
}

The DatabaseUtil class provides some utilities to facilitate the usage of hibernate APIs. It is not shown here for simplicity's sake.

Now, we would like to start a transaction before processing the record, commit it if everything is ok and roll it back when an exception occurs. You could write this transaction handling code directly in the TweetLoader above, but this would pollute your business logic with technical code to handle transactions

In order to introduce cross cutting behavior into the processing workflow, Easy Batch provides listeners that are fired before and after each step. In this case, we need a listener bound to the record processing step. We can implement the RecordProcessorEventListener interface:

TransactionProcessingEventListener.java
public class TransactionProcessingEventListener implements RecordProcessorEventListener {

    private static final Logger LOGGER = Logger.getLogger(TweetLoader.class.getName());

    @Override
    public void beforeProcessingRecord(Object record) {
        DatabaseUtil.getCurrentSession().beginTransaction();
    }

    @Override
    public void afterProcessingRecord(Object record, final Object processingResult) {
        DatabaseUtil.getCurrentSession().getTransaction().commit();
        LOGGER.log(Level.INFO, "Tweet {0} successfully persisted in the database", record);
    }

    @Override
    public void onRecordProcessingException(Object record, Throwable throwable) {
        DatabaseUtil.getCurrentSession().getTransaction().rollback();
    }
}

This listener will start, commit and rollback the transaction when needed. This is a kind of aspect-oriented programming that allows you to benefit from separating business logic from cross cutting concerns.

Heads up!

The implementation of TransactionProcessingEventListener is kept simple for demonstration purpose. In production use case, you should define a commit interval to avoid performance issues of committing the transaction after each record insertion.

You should now register the new record processor and the transaction listener as follows:

public class Launcher {

    public static void main(String[] args) throws Exception {

    //load tweets from tweets.csv
    File tweets = new File(args[0]);

    //Start embedded database server
    DatabaseUtil.startEmbeddedDatabase();
    DatabaseUtil.initializeSessionFactory();

    // Build a batch engine
    Engine engine = new EngineBuilder()
        .reader(new FlatFileRecordReader(tweets))
        .filter(new HeaderRecordFilter())
        .mapper(new DelimitedRecordMapper(Tweet.class, new String[]{"id", "user", "message"}))
        .validator(new BeanValidationRecordValidator<Tweet>())
        .processor(new TweetLoader())
        .recordProcessorEventListener(new TransactionProcessingEventListener())
        .build();

    // Run easy batch engine
    engine.call();

    //Dump tweet table to check inserted data
    DatabaseUtil.closeSessionFactory();
    DatabaseUtil.dumpTweetTable();

    }
}

4. Summary

In this tutorial, you have learned how to use Easy Batch in a typical enterprise requirement to persist data into a database. Note that you have only wrote the TweetProcessor which is your application's logic and have reused exactly the same components that served for another application. This is how flexible Easy Batch's API is to enforce reusability.

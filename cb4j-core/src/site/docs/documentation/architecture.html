Architecture

The next figure illustrates the main CB4J components and gives an overview of the framework architecture.

CB4J has a clearly defined workflow to process CSV flat files. The main components involved in this workflow are the following :

  • Record Parser : responsible for record parsing
  • Record Validator : responsible for applying the declared validation rules
  • Record Mapper : responsible for mapping a record to a java object
  • Record Processor : responsible for applying business processing logic for each record

The workflow used internally by CB4J follows the next algorithm :

  1. The record reader reads the input CSV file line by line until the end of file. If the end of file is reached, the engine instructs the batch reporter to generate the batch report and terminates the execution
  2. The next record is read as a string containing raw data
  3. The record parser checks the record for well formedness : If the record is not well formed (actual size not equal to expected size), then the record is ignored and the engine moves to next one. If the record is well formed, then it is passed to the next step to be parsed
  4. The record parser parses the current record to split it into fields ordered by the index and containing raw data
  5. The record validator applies validation rules for each field. If a validation rule is violated for one field, then the record is rejected. If all fields are valid, then the parsed record is passed to the next step to be mapped to an object
  6. At this point, the record mapper is supplied with a parsed record containing raw data that have passed the declared validation rules. The parsed record is mapped to the declared object . CB4J consider mapping exception as errors that should not pass the processing step. Hence, the engine rejects records that cannot be mapped to objects with typed fields. This is a design choice to minimize unexpected errors during record processing. NB : In a well designed batch, the record mapper should not have any problem in converting string values to typed values if all data was validated at validation step.
  7. This is where the business logic is (finally!) executed. At this point, the record processor is supplied with an object view of the CSV record with validated and typed data.
